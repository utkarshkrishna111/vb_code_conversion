Usage & Operations Guide

Java → Python Migration Orchestrator

A multi-agent AI pipeline that converts Java classes and modules to idiomatic, type-annotated Python — with full test coverage — in three automated steps, each with a mandatory human review gate. This guide covers everything from setting up your first migration to reviewing each stage and troubleshooting failures.

Hub & Spoke Architecture Test-Driven Migration LiteLLM · Claude / OpenAI / Gemini MCP Infrastructure Human Gate Every Step
🏗
Architecture & Design
How the system is structured and why

🎓 Stakeholders The orchestrator uses a Hub & Spoke pattern — one central Hub Agent coordinates a set of specialised agents (spokes), each responsible for a single concern such as understanding Java code, writing tests, or generating Python. No agent does more than one job.

🔧 Tech Team Every agent calls the Claude API (Anthropic) with a dedicated system prompt. The Hub Agent is the sole keeper of migration state; spoke agents are stateless and receive all context they need in their prompt.

🗄️

Preparing Your Java Input
What to provide and how to organise source files

🎓 Stakeholders The migration system reads your existing Java source files directly — no conversion or pre-processing is needed. Simply point the tool at the directory containing your .java files.

🔧 Tech Team The File System MCP scans the source directory recursively for all .java files. Each file is treated as one independent module and processed in sequence.

Supported File Types

ExtensionLanguageNotes
.javaJavaStandard Java source files — classes, interfaces, enums. Each file is treated as one independent module.

Recommended Source Directory Layout

your_java_project/ ├── src/ │ ├── InvoiceCalculator.java ← business logic │ ├── CustomerRepository.java ← data access │ └── ReportFormatter.java ← presentation └── io_samples/ ← optional: historical I/O data ├── InvoiceCalculator_io.json └── CustomerRepository_io.json

Historical I/O Samples (Optional but Recommended)

If you have logs or recorded inputs/outputs from the original Java system, place them as JSON files alongside your source. The Golden Master Agent uses them to create characterisation tests — the strongest guarantee that migrated code behaves identically to the original. 

// io_samples/InvoiceCalculator_io.json
[
  {
    "input": { "items": [{ "qty": 2, "unit_price": 50.00 }], "discount_pct": 0.1, "vat_rate": 0.2 },
    "expected_output": 108.00
  }
]

No I/O samples?
The Golden Master Agent will synthesise representative samples from the Markdown spec. Real data is always better, but synthetic samples still provide meaningful regression coverage.

Step 1 — Discovery, Documentation & Architecture
Understanding the Java module and designing the Python equivalent

🎓 Stakeholders Step 1 answers the question "What does this Java code actually do?" It produces a plain-English Markdown document you can read and a machine-readable architecture blueprint for the Python replacement. A mandatory human review gate always triggers at the end of Step 1.

A
Understand Agent
agents/analyze_and_design/understand_agent.py · Model: claude-sonnet-4-6

Reads the Java source and produces a structured JSON analysis covering all classes, methods, external API calls, control flow (for/while/try-catch), error handlers, and a complexity score (0–1).

AnalysisResult JSON complexity_score: 0.0–1.0

What to check

  • 🔍
    All class and method definitions captured
    The AST Checker (see below) flags any methods found in source but missing from the analysis.
  • 🔍
    Complexity score is reasonable
    0.0 = trivial CRUD. 1.0 = deeply nested logic with multiple checked exceptions and external calls. The human gate always triggers after Step 1 completes.
  • 🔍
    External API calls listed
    Every external dependency (JDBC, JPA, REST clients) must be identified here — these become dependency-injection points in the architecture.
B
Document Agent
agents/analyze_and_design/document_agent.py · Model: claude-sonnet-4-6

Converts the JSON analysis into a Markdown specification covering overview, inputs/outputs table, numbered business rules, data structure tables, error conditions, and migration notes for Java-specific idioms.

{module}_spec.md

What to check

  • 📋
    Business rules are accurate
    Have a domain expert read the numbered list. These rules are the source-of-truth for test generation in Step 2.
  • 📋
    All Java idioms noted
    Null returns, checked exceptions, synchronized blocks, and raw collection types must appear in "Migration Notes".
  • 📋
    Inputs/outputs table complete
    Every public method must have a row showing parameters, types, and return values.
C
Architect Agent
agents/analyze_and_design/architect_agent.py · Model: claude-sonnet-4-6

Designs the Python equivalent: Pydantic/dataclass models, fully-annotated public API signatures, module boundaries, and dependency injection points for all external services.

{module}_architecture.json

What to check

  • 🏛
    No global mutable state
    The Arch Auditor flags this automatically. All shared data must flow through function parameters or injected services.
  • 🏛
    All public functions fully typed
    Every entry in public_api must have a complete signature: fn(arg: Type) -> ReturnType.
  • 🏛
    JDBC/JPA/REST clients become injected abstractions
    e.g. new JdbcTemplate(dataSource) → DatabaseRepository protocol injected into the constructor.

Step 1 Validators

VAL
AST Checker + Architecture Auditor
agents/analyze_and_design/ast_checker.py & arch_auditor.py
ValidatorHow it worksOn failure
AST Checker Regex scan of Java source for all class and method definitions, compared against the Understand Agent output. Also flags null returns, instanceof chains, and synchronized blocks. Issues logged as warnings; pipeline continues (non-blocking) but the Hub logs missing methods.
Arch Auditor Claude LLM review of the architecture JSON — checks for global state, missing type hints, Java-isms (get/set prefixes, null returns, checked exceptions leaking into API), and poor OOP. If passed: false, the auditor issues list is shown at the mandatory Human Gate at the end of Step 1.

⚠️

The Human Gate fires unconditionally after every Step 1 run — there is no threshold or condition. The CLI pauses and displays the spec path, architecture path, complexity score, and any Arch Auditor issues. You must type y or n before the pipeline proceeds.

Step 2 — Test-Driven Development
Generating, auditing, and validating the test suite before any Python code exists

🎓 Stakeholders Step 2 creates the tests before the Python code. This means the Python code is judged against an objective standard derived from the original Java behaviour — not written to pass trivial checks.

🔧 Tech Team Two test agents run in parallel using asyncio.gather() — cutting generation time roughly in half. Their outputs are merged and deduped by the Test Auditor before the Dry Run validates syntax.

‖
Parallel: Functional Tests (2-A) + Golden Master (2-B)
Both agents run simultaneously via asyncio.gather()
Agent 2-A
Functional Tests
functional_tests_agent.py

Writes pytest unit tests from the Markdown spec. Covers every business rule, boundary condition, and edge case. Uses @pytest.mark.parametrize for data-driven cases.

test_{module}_functional.py
Agent 2-B
Golden Master
golden_master_agent.py

Creates characterisation tests from real Java I/O data (or synthetic samples). Asserts exact output match — the strongest guarantee of behavioural equivalence.

test_{module}_golden.py
AUD
Test Auditor
agents/build_test_execute/test_auditor.py

The Test Auditor reviews both test files, removes trivial assertions (e.g. assert True), eliminates over-mocking, fills coverage gaps identified from the spec, and merges them into a single authoritative test file.

test_{module}.py (merged)

What to check in the merged test file

  • 🧪
    At least one test per business rule
    Cross-reference against the numbered list in the Markdown spec. Every rule should map to ≥ 1 test.
  • 🧪
    Boundary conditions covered
    Zero values, empty collections, max values, negative inputs — all must appear.
  • 🧪
    No mocking of core logic
    Only external I/O (file system, DB, HTTP) should be mocked. Business logic must run for real.
  • 🧪
    Golden master tests present
    Look for @pytest.mark.parametrize blocks with inline expected values matching the Java I/O data.
RUN
Dry Run — pytest --collect-only
agents/build_test_execute/dry_run_runner.py → Execution MCP

Runs pytest --collect-only -q on the merged test file. This validates Python syntax and import paths without executing any tests — which would fail because the application code doesn't exist yet.

✅
Expected outcome
All tests are collected (listed) with exit code 0. Syntax errors or bad imports will show here.
⚠️
If dry run fails
The Hub logs the collection errors but continues to Step 3. The Converter Agent will see the test file has syntax issues and should self-correct the tests on retry. If they persist, check the merged test file manually.

1
Converter Agent
agents/generate_code/converter_agent.py · Model: claude-sonnet-4-6

Writes a complete Python module implementing every function in the architecture's public_api. On retry, it receives the full pytest error output and must fix all failing tests.

The Converter also queries the Vector DB for previously solved Java→Python patterns — reducing redundant re-derivation on similar constructs.

What to check in the generated code

  • 🐍
    All public API functions present
    Cross-reference with public_api in the architecture JSON.
  • 🐍
    Full type annotations on every function
    mypy --strict will catch any gaps before pytest runs.
  • 🐍
    No global mutable variables
    Module-level constants (MAX_DISCOUNT: float = 0.3) are fine; mutable dicts/lists at module level are not.
  • 🐍
    Error handling uses specific try/except, not broad catches
    Java's checked exceptions must map to specific Python exception types — no bare except: or except Exception:.

Self-correction loop
The full pytest output is fed back to the Converter Agent with the instruction "Fix ALL failing tests." This repeats up to MAX_RETRIES times (default 3). If all retries are exhausted, the module is escalated to the CLI operator.

Quality Checklist
What to review at each stage before signing off

🔧 Tech Team Use this checklist when manually reviewing output artifacts or when a module fails the pipeline and you're diagnosing the cause.

Step 1 — Analysis & Architecture

  • ☐
    complexity_score is proportionate
    Simple utilities should score < 0.3. Modules with deeply nested loops, multiple checked exceptions, and external API calls should score > 0.6.
  • ☐
    All public methods in analysis.control_flow.methods
    Open the Java file and count public method declarations. Counts must match the analysis output.
  • ☐
    Markdown spec matches actual Java behaviour
    Have a developer familiar with the original system read section 2 (Business Rules). Incorrect rules here cascade into wrong tests and wrong code.
  • ☐
    Architecture has zero global mutable state
    Search the architecture JSON for "global" or "module-level" in design_notes. Any mutable dict/list at module scope is a red flag.
  • ☐
    Every JDBC/JPA/REST dependency is an injected abstraction
    e.g. JdbcTemplate or RestTemplate → protocol class injected into the constructor, listed in dependency_injection.
  • ☐
    No Java getter/setter naming in Python
    getCustomerName() → customer_name property. setCount(n) → direct attribute assignment.

Step 2 — Test Suite

  • ☐
    One test per business rule minimum
    Count numbered rules in the spec. Count def test_ functions covering each. Gaps must be added manually.
  • ☐
    Boundary values tested
    For ApplyDiscount: 0%, 30% (max), 31% (over), and negative inputs. All four should have test cases.
  • ☐
    Exception paths tested
    Every business rule that says "raises X on Y" must have a with pytest.raises(X) test.
  • ☐
    No assert True, assert 1 == 1
    Grep the test file: grep "assert True\|assert 1 == 1" test_*.py — output must be empty.
  • ☐
    Golden master data matches actual Java output
    If you have real Java I/O logs, spot-check 5 parametrize cases against them.
  • ☐
    Dry run passes (all tests collected)
    Run manually: pytest --collect-only output/{module}/test_{module}.py

Step 3 — Generated Python Code

  • ☐
    mypy --strict exits 0
    Run: mypy --strict output/{module}/{module}.py
  • ☐
    All tests pass
    Run: pytest -v output/{module}/test_{module}.py
  • ☐
    Line count is proportionate
    Python should be ~50–70% of the Java line count (less ceremony, same logic). Suspiciously short = likely missing logic.
  • ☐
    No Any types in production paths
    Run: grep "Any" output/{module}/{module}.py — any hits should be justified with a comment.
  • ☐
    Error handling is specific
    No bare except: clauses. Each exception should name the specific type (e.g. except ValueError).
  • ☐
    Dependencies are injected (not hardcoded)
    No hardcoded connection strings, file paths, or API endpoints in the module body.

Test Case Review & Manual Test Authoring
Understanding generated tests, adding your own, and verifying coverage

🔧 Tech Team The generated test file lives at output/{module}/test_{module}.py. You can edit it directly. The Converter Agent re-reads it on every retry, so manual improvements are preserved.

Anatomy of a Generated Test File

# Generated by Java→Python Migration Orchestrator — Step 2
import pytest
from decimal import Decimal
from InvoiceCalculator import LineItem, Invoice, calc_subtotal, apply_discount, calc_total

# ── Fixtures ────────────────────────────────────────────────
@pytest.fixture
def sample_invoice() -> Invoice:
    return Invoice(
        invoice_number="INV-001",
        customer_id="CUST-42",
        items=[LineItem(product_code="SKU-1", quantity=2, unit_price=Decimal("50.00"))],
        discount_pct=0.1,
        vat_rate=0.2,
    )

# ── Functional Tests ─────────────────────────────────────────
def test_calc_subtotal_single_item(sample_invoice):
    """Rule 1: subtotal = sum(qty × unit_price) for all line items."""
    assert calc_subtotal(sample_invoice) == Decimal("100.00")

@pytest.mark.parametrize("pct,expected", [
    (0.0,  Decimal("100.00")),
    (0.1,  Decimal("90.00")),
    (0.3,  Decimal("70.00")),   # maximum allowed
])
def test_apply_discount_valid(sample_invoice, pct, expected):
    assert apply_discount(Decimal("100.00"), pct) == expected

def test_apply_discount_exceeds_max_raises():
    """Rule 2: discount > 30% must raise ValueError."""
    with pytest.raises(ValueError, match="out of range"):
        apply_discount(Decimal("100.00"), 0.31)

# ── Golden Master Tests ──────────────────────────────────────
@pytest.mark.parametrize("items,discount,vat,expected_total", [
    ([(2, Decimal("50.00"))], 0.1, 0.2, Decimal("108.00")),
    ([(1, Decimal("100.00"))], 0.0, 0.2, Decimal("120.00")),
])
def test_golden_master_calc_total(items, discount, vat, expected_total):
    """Exact match against recorded Java system output."""
    inv = Invoice(invoice_number="T", customer_id="T",
                  items=[LineItem(product_code="X", quantity=q, unit_price=p) for q, p in items],
                  discount_pct=discount, vat_rate=vat)
    assert calc_total(inv) == expected_total

Adding Your Own Tests

Add tests directly to output/{module}/test_{module}.py. Follow these conventions to keep them compatible with the pipeline:

  • ✏️
    Use pytest.mark.parametrize for multiple input variants
    Avoids copy-paste test functions that all look the same.
  • ✏️
    Write a one-line docstring referencing the spec rule number
    e.g. """Rule 4: VAT is applied after discount."""
  • ✏️
    Use pytest.raises(ExceptionType, match="pattern")
    The match pattern ensures the right exception message is checked, not just the type.
  • ✏️
    Use Decimal for all monetary values
    Floating-point rounding errors will cause flaky golden master tests. Always use Decimal("0.10") not 0.10.

Checking Test Coverage

# Install coverage plugin (once)
pip install pytest-cov

# Run with coverage report
pytest --cov=output/InvoiceCalculator \
       --cov-report=term-missing \
       output/InvoiceCalculator/test_InvoiceCalculator.py

Aim for ≥ 90% line coverage. Lines marked as MISS in the report indicate logic paths untested — add parametrize cases for them.

Each MCP server is a real stdio-transport MCP process. The Hub Agent spawns all four server subprocesses at startup using an AsyncExitStack and shuts them down cleanly when the migration run ends. You never need to start or stop them yourself — just run python main.py ….

Output Artifacts
What gets generated, where it lives, and what to do with it

Every migrated module produces its own subdirectory under output/:

output/ └── InvoiceCalculator/ ├── InvoiceCalculator.py ← ✅ final Python module ├── test_InvoiceCalculator.py ← ✅ merged test suite ├── InvoiceCalculator_spec.md ← 📋 Markdown specification └── InvoiceCalculator_architecture.json ← 🏛 architecture blueprint output/ └── migration_state.json ← 🗂 global migration log (all modules)

Artifact Descriptions

{module}.py
The migrated Python module. Fully type-annotated, passes all tests. This is the deliverable — copy it to your target repository.
test_{module}.py
Merged functional + golden master test suite. Keep this in the target repo alongside the module. These are your regression tests going forward.
{module}_spec.md
Human-readable functional specification derived from the Java source. Useful for code review and as permanent documentation of the module's business rules.
{module}_architecture.json
Machine-readable architecture blueprint (dataclasses, public API signatures, module boundaries). Used internally by Steps 2 and 3; useful for debugging unexpected designs.

migration_state.json

The Hub Agent persists its global state after every status change. If a run is interrupted, the state file shows exactly which modules completed and which are pending.

Troubleshooting
Common problems and how to resolve them
🔧 Tech Team
SymptomLikely causeFix
anthropic.AuthenticationError ANTHROPIC_API_KEY missing or invalid Check .env — key must start with sk-ant-
Step 3 exhausts all retries Architecture/test mismatch; or very unusual Java idiom 1. Check generated Python manually. 2. Add a similar solved pattern to Qdrant. 3. Edit the architecture JSON to simplify the API.
mypy errors persist after retries Incorrect return type in architecture public_api Edit {module}_architecture.json, fix the signature, delete the generated .py, re-run Step 3 only by calling ConverterAgent directly.
Golden master tests fail with wrong values The I/O samples in io_samples/ are incorrect or the Java logic was already buggy Verify samples against the actual Java system output. Update the JSON, delete the test file, and re-run Step 2.
No Java files found Wrong source directory or unexpected file extensions Check FileSystemServer.find_java_files() — it scans recursively for .java files. Verify the source directory path.
ModuleNotFoundError when running pytest Test imports module by name but Python can't find it Run: PYTHONPATH=output/{module} pytest output/{module}/test_{module}.py
Qdrant connection refused Qdrant not running Start Docker: docker run -p 6333:6333 qdrant/qdrant — or unset QDRANT_URL to use in-memory mode
GitHub 422 on PR creation Token scope missing or repo name wrong Token needs repo scope. GITHUB_REPO must be owner/repo (not a URL)

Re-running a Failed Module

The current version processes each module from scratch. To re-run a specific module without repeating the whole batch:

# Point source_dir at a single-file subdirectory
mkdir tmp_retry && cp your_java_project/InvoiceCalculator.java tmp_retry/
python main.py tmp_retry/ --out output/ --retries 5
rm -rf tmp_retry

Debug Logging

Set LOG_LEVEL=DEBUG in .env to see every Claude API call, token usage, and tool result:

LOG_LEVEL=DEBUG python main.py sample_java/ 2>&1 | tee migration.log

Machine Setup — Step by Step
Everything you need to install and configure before running your first migration
🔧 Tech Team

Step 1 — Install Python 3.11+

Check your current version and install if needed:

# Check installed version
python3 --version
# Must print Python 3.11.x or higher

# macOS (Homebrew)
brew install python@3.11

# Ubuntu / Debian
sudo apt-get install python3.11 python3.11-venv

Step 2 — Clone the Project

git clone <repo-url> vb_code_conversion
cd vb_code_conversion

Step 3 — Create a Virtual Environment

python3 -m venv .venv
source .venv/bin/activate      # Windows: .venv\Scripts\activate
⚠️
Always activate the venv first
All subsequent commands — including python main.py, pytest, and mypy — must run inside the activated virtual environment. The MCP servers are spawned as subprocesses and inherit the same environment.

Step 4 — Install Dependencies

pip install -r requirements.txt

This installs: anthropic, litellm, mcp, pydantic, rich, typer, qdrant-client, PyGithub, aiofiles, python-dotenv, mypy, pytest.

Step 5 — Create your .env file

cp .env.example .env
nano .env        # or any editor

Minimum required content:

ANTHROPIC_API_KEY=sk-ant-…your key here…

Optional extras:

GITHUB_TOKEN=ghp_…token…
GITHUB_REPO=myorg/myrepo
QDRANT_URL=http://localhost:6333
MAX_RETRIES=3
# Human review gates are mandatory after every step — no threshold needed

Step 6 — (Optional) Start Qdrant for Translation Memory

Skip this step if you don't need persistent translation memory — the pipeline works fine in in-memory mode.

docker run -d -p 6333:6333 --name qdrant qdrant/qdrant

Step 7 — Verify Installation

python main.py --help

Expected output:

Usage: main.py [OPTIONS] SOURCE_DIR

  Java → Python Migration Orchestrator

Arguments:
  SOURCE_DIR  Directory containing Java source files  [required]

Options:
  --out      TEXT     Output directory  [default: ~/output]
  --retries  INTEGER  Max self-correction retries  [default: 3]
  --help             Show this message and exit.

Step 8 — Run a Smoke Test with the Sample

python main.py sample_java/ --retries 2

You should see progress panels for each of the 3 steps. A successful run ends with the green completion panel showing Completed: 1.

Converting One Package — End-to-End Walkthrough
A concrete example using InvoiceCalculator.java — what happens at each stage and when to review

🎓 Stakeholders This section walks you through what happens when you migrate a single Java module. Each sub-section shows what the system produces, what you should review, and who calls what next.

Launch the Migration

python main.py path/to/java_project/ --retries 3

The Hub Agent starts all MCP servers, scans for .java files, and begins processing each module in sequence. Output goes to ~/output by default.

1A
Stage 1-A: Understand the Java Code
Called by: hub_agent.py → _analyze_and_design() → UnderstandAgent().run()

The UnderstandAgent reads InvoiceCalculator.java and produces a structured JSON analysis — all classes, methods, external API calls, and a complexity score.

👁️
Human Review Point — Understand Output
The analysis is not written to a file you can read directly, but you will see the complexity score printed on screen. If the score seems wrong (e.g. a trivial module scores 0.9), the cause is usually unusual Java idioms. You can inspect the raw analysis by adding a console.print(analysis.model_dump_json(indent=2)) call in hub_agent.py:_analyze_and_design() temporarily.

What to call next: Nothing — the Hub Agent proceeds automatically to Stage 1-B. The AST Checker runs silently and logs any missing methods as warnings.

1B
Stage 1-B: Write the Markdown Spec
Called by: hub_agent.py → _analyze_and_design() → DocumentAgent().run()

The DocumentAgent converts the JSON analysis into a human-readable Markdown specification.

output/InvoiceCalculator/InvoiceCalculator_spec.md
Plain-English specification: business rules, inputs/outputs table, error conditions, migration notes. Written to disk by the File System MCP.
📋
Human Review Point — Markdown Spec
Open output/InvoiceCalculator/InvoiceCalculator_spec.md in your editor.
Check: (1) the numbered business rules are accurate, (2) all Java-specific idioms are noted (null returns, checked exceptions, synchronized blocks), (3) the inputs/outputs table has a row for every public method.
If a rule is wrong, edit the spec file directly — the Test Auditor in Step 2 reads this spec to fill coverage gaps.

What to call next: Nothing — the Hub Agent proceeds automatically to Stage 1-C. If you edited the spec, re-save and press Enter in the terminal (the pipeline has already moved on; edits are picked up by Step 2 which re-reads the file from disk).

1C
Stage 1-C: Design the Python Architecture
Called by: hub_agent.py → _analyze_and_design() → ArchitectAgent().run()

The ArchitectAgent produces a detailed JSON blueprint: dataclasses, type-annotated public API signatures, dependency injection points, and module boundaries.

output/InvoiceCalculator/InvoiceCalculator_architecture.json
Architecture blueprint — dataclasses, public_api list with full signatures, design_notes, dependency_injection list.
🏛️
Mandatory Human Gate — Step 1 Review
After the Arch Auditor runs, the pipeline always pauses and shows you:

• Spec path and architecture path
• Complexity score
• List of auditor issues (if any)

You are asked: "Approve spec & architecture? Proceed to test generation? [y/n]"

Type y to proceed to Step 2.
Type n to mark the module FAILED — fix the Java source, spec, or architecture JSON and re-run.

To edit before approving: open ~/output/InvoiceCalculator/InvoiceCalculator_architecture.json, fix the public_api entries or design_notes, save, then type y.

What to call next (if you reject): Edit the architecture JSON or the Java source, then re-run with python main.py tmp_single/ pointing to just this module.

2
Stage 2: Generate & Audit Tests (parallel)
Called by: hub_agent.py → _build_test_execute() → asyncio.gather(FunctionalTestsAgent, GoldenMasterAgent) → TestAuditor → DryRunRunner

Two test agents run simultaneously, their outputs are merged by the Test Auditor, and the merged file is validated with pytest --collect-only.

output/InvoiceCalculator/test_InvoiceCalculator.py
Merged test suite — functional tests from the spec + golden master tests from I/O data. Validated by dry run before Step 3.
🧪
Human Review Point — Test Suite
Open output/InvoiceCalculator/test_InvoiceCalculator.py
Check: (1) at least one def test_… per numbered business rule in the spec, (2) boundary values covered, (3) pytest.raises for error conditions, (4) no assert True placeholders.

You can add or edit tests directly in this file. Changes are picked up by the Converter Agent in Step 3.

What to call next: The pipeline pauses at the mandatory Step 2 Human Gate — review the test file and type y to approve. The dry run result is also shown; any collection errors are listed.

3
Stage 3: Convert & Self-Correct
Called by: hub_agent.py → _generate_code() → ConverterAgent → MypyChecker → PytestRunner (closed loop, up to MAX_RETRIES)

The Converter generates Python code, mypy validates types, pytest validates behaviour. On failure, the error output is fed back to the Converter for self-correction.

output/InvoiceCalculator/InvoiceCalculator.py
Final migrated Python module — fully type-annotated, all tests pass, ready for your target repository.
✅
Happy path — all tests pass
Console shows: mypy PASS → pytest N passed → ✓ InvoiceCalculator — all tests pass!
The pipeline then pauses at the mandatory Step 3 Human Gate. Type y to save the pattern to translation memory and create the GitHub PR (if configured).
🔄
Retry budget exhausted — Escalation to CLI
If all retries fail, the terminal pauses with the last error and a prompt:
"Press Enter to acknowledge and continue to the next module"

At this point you can:
• Press Enter to skip and continue with other modules
• Open the generated .py file, fix it manually, then re-run Step 3 only by calling:
python -c "import asyncio; from agents.generate_code.converter_agent import ConverterAgent; ..."
(see Troubleshooting for the full recipe)

Making the Tool Interactive
Options for adding a UI, step-by-step wizard, or dashboard to the pipeline

The pipeline currently runs as a headless CLI. Three approaches add progressively richer interactivity — choose based on your team's comfort and the project's scope.

Option A — Streamlit Dashboard (Recommended for teams)

A Streamlit app can drive the Hub Agent step-by-step, showing artifacts in real time and collecting human-gate decisions through UI forms instead of terminal prompts.

📊
Module Status Board
Live grid showing each Java file → status (Analyzing / Documented / Converting / Done / Failed). Colour-coded, refreshes every 2 s via st.rerun().
👁️
Inline Spec & Arch Viewer
Click a module row to open its spec.md and architecture.json side-by-side. Renders Markdown natively with st.markdown().
⚠️
Human Gate Dialog
Replace rich.Confirm.ask() with a st.dialog popup. The Hub Agent blocks on an asyncio.Event until the reviewer clicks Approve or Reject.
📝
Editable Spec & Tests
Render spec.md and test file in st.text_area() with a Save button. Write changes back via the File System MCP before proceeding.
# Quick start — add streamlit to requirements and create app.py
pip install streamlit

Option B — Step-by-Step CLI Wizard

Keep the terminal but make each stage a confirmed step. Replace the automated for module_state in modules loop with an interactive prompt that shows the artifact before continuing.

💡
How to implement it
In agents/hub_agent.py, after each await self.fs.write_text(spec_path, spec.content), add a rich.Confirm.ask(f"Step 1-B done. Open {spec_path} to review. Continue to Step 1-C?"). This turns the pipeline into a step-by-step wizard with a single-line change per stage.

Option C — Jupyter Notebook Driver

For exploratory or one-off migrations, a notebook gives cell-by-cell control with full Python flexibility.

# Cell 1 — Setup
import asyncio
from agents.analyze_and_design.understand_agent import UnderstandAgent
from agents.analyze_and_design.document_agent import DocumentAgent
from agents.analyze_and_design.architect_agent import ArchitectAgent
from infrastructure.clients.filesystem_client import FilesystemClient

# Cell 2 — Read Java source
java_source = open("path/to/InvoiceCalculator.java").read()

# Cell 3 — Understand
analysis = UnderstandAgent().run(java_source, "InvoiceCalculator", "InvoiceCalculator.java")
print(f"Complexity: {analysis.complexity_score:.2f}")

# Cell 4 — Review analysis, then generate spec
spec = DocumentAgent().run(analysis, "output/InvoiceCalculator_spec.md")
display(Markdown(spec.content))      # renders in notebook

# Cell 5 — Edit the spec in the cell above if needed, then architect
design = ArchitectAgent().run(analysis, spec, "output/InvoiceCalculator_arch.json")
